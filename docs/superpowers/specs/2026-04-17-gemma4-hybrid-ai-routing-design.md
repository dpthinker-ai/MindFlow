# Gemma 4 Hybrid AI Routing Design

## 背景

当前仓库里的 AI 能力分成两条彼此割裂的路径：

- 云侧任务主要集中在 `AiServiceClient`
- 端侧任务主要集中在 `OnDeviceAiClient`

现状有三个核心问题：

1. 正文润色、主题提取、标签分类等高频编辑任务仍以云侧为主，Gemma 4 没有进入主链路。
2. 图谱任务没有真正接入端侧 concept graph 流程，导致端侧模型能力没有被用于知识点抽取、归并、关系推断。
3. `preferOnDevice` 只作用于少数 Flow / 本地知识维护链路，没有形成统一的任务路由和回退机制。

这不是“Gemma 4 能力不足”的问题，而是“任务路由、prompt 设计、质量判定、回退策略”没有围绕端侧优先重新设计。

## 目标

1. 建立统一的双通路 AI 架构，默认端侧优先，云侧只在必要时兜底。
2. 让正文润色、主题提取、标签分类、图谱知识点抽取与关系生成全部进入统一路由层。
3. 为 Gemma 4 提供按任务定制的端侧 prompt，而不是直接复用云侧 prompt。
4. 为 `自动` 模式定义明确的结构判定、质量判定、回退条件和埋点。
5. 保留用户可理解的全局执行策略：`自动`、`仅端侧`、`仅云侧`。

## 非目标

1. 不在本轮统一所有 AI 功能的底层推理引擎实现细节。
2. 不在本轮引入任务级临时切换入口，只提供全局执行策略。
3. 不在本轮追求端侧完全替代云侧，云侧仍然保留兜底职责。
4. 不在本轮把所有旧 prompt 一次性重写到最优，只先覆盖当前最关键的一组任务。
5. 不在 spec 阶段直接进入代码实现或发布。

## 产品策略

### 全局执行模式

新增统一的 AI 执行模式：

- `自动`
- `仅端侧`
- `仅云侧`

行为定义如下：

1. `自动`
   默认端侧优先执行。
   如果端侧输出通过结构判定和质量判定，则直接采用。
   如果端侧失败或质量不足，则自动回退云侧。

2. `仅端侧`
   只允许端侧执行。
   若端侧失败，则直接返回失败状态或低质量提示，不自动补云侧。

3. `仅云侧`
   直接走云侧，不触发端侧。

### 适用任务范围

第一阶段统一接入以下任务：

1. 正文润色
2. 主题提取
3. 标签 / 分类提取
4. 图谱知识点抽取
5. 图谱知识点归并
6. 图谱局部关系生成

## 架构设计

### 统一入口

新增统一任务路由层，例如：

- `AiTaskRouter`
- `AiTaskRequest`
- `AiTaskResult`
- `AiExecutionMode`
- `AiProvider` (`on_device` / `cloud`)

所有业务调用只依赖 router，不再直接依赖云侧或端侧 client。

### Provider 分层

底层仍保留两类 provider：

1. `OnDeviceAiProvider`
   基于 Gemma 4 端侧模型，负责端侧任务执行。

2. `CloudAiProvider`
   基于现有云侧 AI 能力，负责兜底或显式云侧执行。

router 负责：

- 读取全局执行模式
- 按任务类型选择 provider 顺序
- 执行结构判定和质量判定
- 决定是否回退
- 记录执行元信息

### 依赖边界

改造完成后，以下业务层不再直接依赖 `AiServiceClient`：

- 主题提取器
- 正文润色调用点
- 标签 / 分类调用点
- 图谱 planner

它们统一依赖 `AiTaskRouter`。

`OnDeviceAiClient` 则从“Flow 专用接口”扩展成“通用端侧任务接口”。

## 任务模型

### 建议任务类型

统一任务类型枚举至少包括：

- `POLISH_CONTENT`
- `EXTRACT_TOPIC`
- `EXTRACT_TAGS`
- `CLASSIFY_CATEGORY`
- `GRAPH_EXTRACT_CONCEPTS`
- `GRAPH_CANONICALIZE_CONCEPTS`
- `GRAPH_GENERATE_RELATIONS`

### 统一响应模型

每类任务返回值分为两层：

1. `payload`
   任务本身结果，例如润色文本、主题、标签、图谱节点边。

2. `meta`
   执行过程信息，例如：
   - `providerUsed`
   - `fallbackOccurred`
   - `fallbackReason`
   - `latencyMs`
   - `qualitySignals`

业务层默认只消费 `payload`，但诊断和埋点可以访问 `meta`。

## 端侧 Prompt 设计

Gemma 4 不直接复用云侧 prompt，而是按任务单独设计。

### 正文润色

目标：

- 保留原意
- 保留原始语气
- 做最小充分修改

约束：

- 不新增原文没有的信息
- 不扩写观点
- 优先修病句、重复、跳跃表达
- 输出自然中文，避免模板腔

输出结构：

- `polishedText`
- `changeSummary`

### 主题提取

目标：

- 输出一个短而可检索的主题短语

约束：

- 禁止使用“记录”“想法”“学习”“随想”等空泛主题
- 禁止直接输出整句正文

输出结构：

- `topic`
- `confidence`
- `whyThisTopic`

### 标签 / 分类提取

目标：

- 先求少量高密度标签，再求稳定主分类

输出结构：

- `primaryCategory`
- `tags[]`
- `discardedCandidates[]`

### 图谱任务

图谱不使用单个大 prompt 直接生成整图，而是拆成三类端侧 prompt：

1. 知识点候选抽取
2. 同义 / 近义归并与标准化
3. 中心点局部邻接关系生成

这样可以降低 Gemma 4 在一次推理里同时完成抽点、归并、关系推断所带来的不稳定性。

## 自动回退与质量判定

### 第一层：结构判定

以下情况直接视为端侧失败，可触发 `自动` 模式下的云侧回退：

1. 空结果
2. schema / JSON 非法
3. 超时
4. 关键字段缺失
5. 图谱节点或边数量为零且不符合该任务最小预期

### 第二层：任务质量判定

结构合法后，还需按任务类型评估最低质量。

#### 正文润色

低质量信号：

- 修改幅度极低但原文明显有病句
- 输出明显模板化
- 输出长度异常收缩
- 引入不在原文中的新信息

#### 主题提取

低质量信号：

- 主题过泛
- 主题过长
- 只是机械复述正文

#### 标签 / 分类

低质量信号：

- 标签过少
- 大量宽泛词
- 重复率高

#### 图谱知识点抽取 / 归并

低质量信号：

- 候选知识点过散
- 标准化失败导致重复节点过多
- 有效标准节点太少

#### 图谱关系生成

低质量信号：

- 边太少
- 关系词高度重复
- 解释空泛
- 孤点率过高

## 图谱链路设计

### 设计原则

图谱默认端侧优先，但采用“小流水线”而不是“一次生成整图”。

### 端侧优先流水线

1. 候选知识点抽取
2. 标准化归并
3. 默认中心点选择
4. 中心点局部邻接关系生成
5. 单一 snapshot 落盘

### Step 1: 候选知识点抽取

输入来源：

- 已沉淀知识点
- 最近活跃笔记
- 当前中心点附近的历史 source

端侧模型只负责抽候选知识点，不直接生成整图。

### Step 2: 标准化归并

将同义、近义、不同表述合并成标准节点，并保留 alias 信息。

### Step 3: 默认中心点选择

默认优先级：

1. 最近活跃知识点
2. 当前上下文中连接度最高的知识点

中心点选择尽量走确定性规则，不交给模型自由决定。

### Step 4: 局部关系生成

Gemma 4 只围绕“中心点 -> 候选邻居”生成局部关系。

每条关系输出：

- `relationType`
- `reasonLine`
- `confidence`

用户点击新节点成为中心点后，再为该中心点执行同样的一层关系生成。

### Step 5: Snapshot 落盘

前端只消费单一 snapshot，不再自行补边、补点、重选子图。

snapshot 至少包含：

- `nodes`
- `edges`
- `centerNodeId`
- `generatedBy`
- `qualitySignals`

### 分阶段回退

图谱回退不是只能整条链路切到云侧。`自动` 模式允许按阶段回退：

- 抽点失败，只回退抽点
- 归并失败，只回退归并
- 关系生成失败，只回退关系生成

这样云侧仍是兜底，但不会重新夺回整条主路径。

## 设置层设计

现有设置仓储存在分裂：

- `AiSettingsRepository`
- `OnDeviceModelSettingsRepository`

新设计里，用户只面对一个统一 AI 设置模型，至少包含：

- `executionMode`
- `onDeviceModelLabel`
- `cloudProviderEnabled`
- `lastSuccessfulProviderByTask`

用户不需要理解每个功能各自连了哪一套 client。

## 埋点与诊断

每次任务都记录统一执行元信息：

- `taskType`
- `executionMode`
- `providerUsed`
- `latencyMs`
- `fallbackOccurred`
- `fallbackReason`

按任务补充质量信号：

- 润色：输出长度变化、修改密度、低质量命中项
- 主题：主题长度、泛词命中、confidence
- 标签：标签数、重复率、宽泛词比例
- 图谱：节点数、边数、孤点率、平均关系置信度

图谱额外记录失败阶段：

- `concept_extraction_failed`
- `canonicalization_failed`
- `relation_generation_failed`
- `schema_validation_failed`
- `quality_gate_failed`

## 错误处理

### 用户可见行为

1. `仅端侧` 模式失败时，明确告诉用户端侧任务失败，不静默补云侧。
2. `自动` 模式发生回退时，用户界面不必每次显式打扰，但系统需要保留可诊断记录。
3. 图谱在无有效关系时，应显示明确空状态，不显示坏掉的空网图。

### 内部处理

1. 结构失败和质量失败必须区分记录。
2. 云侧回退必须携带明确原因。
3. 相同任务的连续端侧失败应可用于后续策略调优。

## 测试与验证

### 单元测试

至少覆盖：

1. 三种执行模式的路由命中
2. `自动` 模式的结构失败回退
3. `自动` 模式的质量失败回退
4. 端侧成功时不触发云侧
5. 图谱阶段性回退逻辑

### 小样本基准集

建立一组真实笔记样本，对比三种模式下的结果：

- 仅端侧
- 仅云侧
- 自动

重点观察：

- 输出稳定性
- 质量差异
- 回退频率

### 真实使用观察

先在 `自动` 模式下做一轮真实使用验证，重点观察：

- 端侧命中率
- 云侧回退率
- 不同任务的失败分布
- 图谱孤点率是否明显下降

## 迁移顺序

建议分三阶段落地：

1. 建立统一任务路由层和执行模式设置
2. 先接正文润色、主题提取、标签 / 分类
3. 最后接图谱小流水线

原因：

- 高频编辑任务更容易验证端侧质量
- 图谱链路更复杂，适合作为第二阶段收口

## 风险与对策

### 风险 1：端侧质量波动大

对策：

- 每类任务单独设质量门槛
- 自动模式允许回退
- 用埋点收敛阈值

### 风险 2：图谱端侧延迟过高

对策：

- 只做中心点一层局部生成
- 分阶段执行，不生成全图

### 风险 3：设置复杂度上升

对策：

- 用户只看到三挡模式
- 不增加任务内临时切换入口

### 风险 4：旧调用点迁移不完整

对策：

- 路由层接入后，逐步收敛直接依赖云侧 client 的调用点
- 在 `AppContainer` 层禁止新增绕过 router 的新依赖

## 成功标准

满足以下条件时，视为这轮设计成功：

1. 正文润色、主题提取、标签 / 分类、图谱任务都能通过统一 router 执行。
2. `自动` 模式下，大多数高频任务默认命中端侧。
3. 端侧失败时，云侧回退行为可解释、可追踪。
4. 图谱任务不再依赖一次性整图生成，而是改为局部知识点流水线。
5. 团队能够通过埋点和测试清楚判断问题是出在路由、prompt、模型还是阈值。
