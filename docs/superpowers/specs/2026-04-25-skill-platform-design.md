# MindFlow Skill Platform Design

Date: 2026-04-25  
Status: Draft for implementation planning

## Purpose

MindFlow should evolve from page-specific AI features into a skill-driven capability platform.

The immediate driver is review chat, but the same platform must later support:

- history chat
- concept graph
- dynamic visual cards
- future daily/home cards
- HTML/JS visualizations generated by different skills

This design is based on the Edge Gallery mechanism study in:

- `docs/research/2026-04-25-edge-gallery-mechanism-study.md`

The key decision is to borrow Edge Gallery's skill architecture, not copy its app structure.

## Decisions

1. First version supports MindFlow internal skills only.
2. JS handles visualization and orchestration. Native handles data queries and system capabilities.
3. Skill output is HTML/WebView-first.
4. First integration target is review chat.
5. Review chat uses mixed skill invocation: automatic by default, with manual skill chips available.
6. Normal users see short execution status. Debug mode can show a collapsible execution trace.
7. JS skills are allowed to use network by default.
8. Official skills should support remote updates. First version may be bundled.
9. Remote official skills must be signature verified.
10. Permissions use a mixed model: coarse manifest declarations plus fine-grained runtime bridge APIs.

## Non-Goals For The First Version

- No third-party community skill marketplace.
- No user-imported local skills.
- No arbitrary native API exposure to JS.
- No migration of today cards in the first phase.
- No complete rewrite of graph internals before the review chat skill path is proven.

## Architecture Overview

The platform has six layers:

1. `SkillRegistry`
2. `SkillPlanner`
3. `SkillRuntime`
4. `NativeToolBridge`
5. `SkillResult`
6. Scenario hosts such as review chat, concept graph, and future cards

Edge Gallery baseline:

```text
User message
  -> LiteRT-LM conversation with tools
  -> model calls loadSkill
  -> model calls runJs or runIntent
  -> AgentTools sends AgentAction through actionChannel
  -> AgentChatScreen executes hidden WebView or native intent
  -> tool result returns to model
  -> model finishes final text response
  -> UI appends image/webview outputs
```

MindFlow first version keeps the same product shape but does tool calling at the app layer so cloud and on-device providers behave the same:

```text
User request
  -> Scenario host
  -> SkillPlanner creates SkillInvocation
  -> SkillRegistry loads SKILL.md and permission manifest
  -> SkillRuntime executes JS in hidden WebView
  -> JS calls NativeToolBridge for private local data
  -> SkillRuntime returns normalized SkillResult
  -> Scenario host either uses SkillResult directly or asks model to summarize it
  -> Scenario host appends typed chat messages
```

This means the user experience stays Edge Gallery-like, while the implementation does not depend on LiteRT-LM tool calling being available in every provider.

### `SkillInvocation`

Every skill run should be represented as an explicit invocation object:

```json
{
  "skillId": "history-query",
  "scriptName": "index.html",
  "data": {
    "intent": "categories",
    "query": "帮我看一下所有的记录，帮我总结一下它们都是什么类型的",
    "timeScope": {"type": "all_time"},
    "needsCard": true
  },
  "modelPass": "summarize_compact_result"
}
```

`modelPass` controls whether the skill result is final:

- `none`: use `SkillResult.result` directly.
- `summarize_compact_result`: pass only compact structured result to the model.
- `analyze_records`: pass curated batches or summaries to the model for deeper analysis.

Counts, date ranges, and coverage metadata always come from NativeToolBridge, never from the model.

## Skill Package

First-party skills should use a package structure close to Edge Gallery:

```text
skills/
  history-query/
    SKILL.md
    scripts/
      index.html
    assets/
      ...
    mindflow.skill.json
```

Edge Gallery uses `SKILL.md` as the source of truth for model-visible skill identity and instructions.

MindFlow should keep that shape:

- `SKILL.md` contains `name`, `description`, and execution instructions.
- `scripts/index.html` exposes the JS entry function.
- `assets/` contains local HTML/CSS/JS/image files served through WebView asset loading.
- `mindflow.skill.json` is a MindFlow-only permission supplement, not a replacement for `SKILL.md`.

### `SKILL.md`

The first `history-query` skill should look close to Edge Gallery's JS skill instructions:

```markdown
---
name: history-query
description: Query, count, summarize, and visualize MindFlow historical notes.
---

# History Query

## Instructions

Use this skill when the user asks about their MindFlow records, including counts,
lists, categories, timelines, or summaries over historical notes.

Call `runJs` with:

- script name: `index.html`
- data: a JSON string with:
  - `intent`: one of `count`, `list`, `categories`, `timeline`, `analysis`
  - `timeScope`: the user requested time scope, or `all_time` if they ask for all records
  - `query`: the original user question
  - `needsCard`: whether a visual card would help

Do not invent counts. Use bridge results for counts, date ranges, and record coverage.
Only include source record links if the user explicitly asks for links or original records.
```

This mirrors Edge Gallery's `SKILL.md` responsibility: tell the model exactly when to use the skill and exactly what JSON to pass into `runJs`.

### `mindflow.skill.json`

MindFlow still needs one extra file because private app data requires explicit permissions.

Initial schema:

```json
{
  "id": "history-query",
  "name": "History Query",
  "description": "Query and summarize MindFlow historical notes.",
  "version": "1.0.0",
  "entry": "scripts/index.html",
  "output": ["text", "webview"],
  "permissions": ["history"],
  "nativeApis": [
    "history.query",
    "history.count",
    "history.timeline",
    "history.categories",
    "history.getRecords",
    "note.open"
  ],
  "remote": {
    "signatureRequired": true
  }
}
```

The manifest uses coarse permissions for product review and fine-grained API names for runtime enforcement. It should not be shown to the model unless a debugging tool needs it.

`model` and `network` permissions are future extensions. They are not part of the first `history-query` implementation because the first milestone must prove the local history facts are correct before giving JS skills broader capabilities.

## JS Runtime Contract

Each JS skill must expose the same outer function shape as Edge Gallery:

```javascript
window["ai_edge_gallery_get_result"] = async (data, secret) => {
  try {
    const input = JSON.parse(data || "{}");
    const result = await window.mindflowSkill.run(input, window.mindflowBridge);
    return JSON.stringify(result);
  } catch (error) {
    return JSON.stringify({ error: String(error?.message || error) });
  }
};
```

The internal implementation can use a MindFlow helper:

```javascript
window.mindflowSkill = {
  async run(input, bridge) {
    return {
      result: "short plain text summary",
      webview: {
        url: "assets/history-card.html",
        iframe: false,
        aspectRatio: 1.333
      }
    };
  }
};
```

The function returns a stringified JSON object. This is intentionally the same contract Edge Gallery documents for JS skills.

## Skill Result Protocol

Skill results should stay close to Edge Gallery and use these raw fields:

- `result`
- `error`
- `image`
- `webview`

Raw JS result:

```json
{
  "result": "human-readable short result",
  "webview": {
    "url": "assets/history-card.html",
    "iframe": false,
    "aspectRatio": 1.333
  }
}
```

Failure result:

```json
{
  "error": "Query timed out"
}
```

The MindFlow host can normalize this into an internal result with `status` and `debug`, but that is host metadata, not the JS skill contract:

```json
{
  "status": "succeeded",
  "skillId": "history-query",
  "result": "human-readable short result",
  "webview": {
    "url": "https://appassets.androidplatform.net/skills/history-query/assets/history-card.html",
    "iframe": false,
    "aspectRatio": 1.333
  },
  "debug": {
    "nativeCalls": [{"name": "history.categories", "durationMs": 42}],
    "recordCoverage": {"matchedCount": 135, "processedCount": 135, "complete": true}
  }
}
```

Do not return raw HTML strings as the primary card payload. Follow Edge Gallery's model: serve local files through a stable WebView asset URL, then insert a WebView message into the chat stream.

## Content Presentation Model

This section follows the observed Edge Gallery model instead of inventing a separate block system.

Edge Gallery uses a chat stream made of typed messages:

- `ChatMessageText`
- `ChatMessageImage`
- `ChatMessageWebView`
- `ChatMessageInfo`
- `ChatMessageCollapsableProgressPanel`

Relevant files:

- `AgentChatScreen.kt`
- `ChatPanel.kt`
- `ChatMessage.kt`
- `Types.kt`

MindFlow should mirror this shape with its own domain names rather than introducing a different rendering model.

### Message Types To Adopt

Initial MindFlow equivalents:

```text
ReviewChatTextMessage
ReviewChatWebViewMessage
ReviewChatImageMessage
ReviewChatInfoMessage
ReviewChatProgressMessage
ReviewChatRecordRefMessage
```

This maps directly to Edge Gallery:

```text
ChatMessageText                  -> ReviewChatTextMessage
ChatMessageWebView               -> ReviewChatWebViewMessage
ChatMessageImage                 -> ReviewChatImageMessage
ChatMessageInfo                  -> ReviewChatInfoMessage
ChatMessageCollapsableProgressPanel -> ReviewChatProgressMessage
```

`ReviewChatRecordRefMessage` is MindFlow-specific because our product has first-class note links. It should only appear when the user asks for source links.

### How Text And WebView Coexist

Edge Gallery does not force a JS skill result to replace the model's text response.

Observed behavior in `AgentChatScreen.kt`:

- model response completes
- if `agentTools.resultImageToShow` exists, UI appends an image message
- if `agentTools.resultWebviewToShow` exists, UI appends a WebView message

MindFlow should follow the same composition model:

```text
assistant text answer
optional skill WebView result
optional record references when requested
```

This means an HTML card is a companion result in the message stream, not the only answer.

### HTML/WebView Card Rules

Edge Gallery's `CallJsSkillResultWebview` contains:

```kotlin
val url: String?
val iframe: Boolean?
val aspectRatio: Float?
```

MindFlow should start with the same minimal contract:

```json
{
  "url": "https://appassets.androidplatform.net/skills/history-query/assets/card.html",
  "iframe": false,
  "aspectRatio": 1.333
}
```

For the first version, do not add a separate large card schema. Keep it close to Edge Gallery:

- `result` for text
- `image` for base64 image
- `webview` for visual card
- `error` for failures

MindFlow-specific additions such as `debug` can be optional metadata, not part of the minimum skill result.

## Model And Skill Interaction Pipeline

This pipeline should also stay close to Edge Gallery.

Observed Edge Gallery flow:

```text
User message
  -> LiteRT-LM conversation with tools
  -> model calls loadSkill
  -> model calls runJs or runIntent
  -> AgentTools sends AgentAction through actionChannel
  -> AgentChatScreen executes JS or intent
  -> tool result returns to model
  -> model finishes final text response
  -> UI appends image/webview outputs after generation
```

MindFlow first version should use the same conceptual flow, with one practical adaptation:

- Edge Gallery uses LiteRT-LM native tool calling.
- MindFlow can initially run skill orchestration at the app layer for both cloud and on-device providers.

This avoids coupling the first version to LiteRT-LM tool calling while still preserving the same product mechanism.

### Provider Execution Modes

MindFlow should expose one logical pipeline to the product layer:

```text
plan -> load skill -> run JS -> get bridge-backed result -> optional model pass -> render messages
```

The provider difference is only inside the implementation:

```text
Cloud model
  -> app builds SkillInvocation
  -> app runs skill
  -> app sends compact SkillResult to cloud model if modelPass is needed
  -> cloud model returns final text

On-device model without tool calling
  -> app builds SkillInvocation
  -> app runs skill
  -> app sends compact SkillResult to local model if modelPass is needed
  -> local model returns final text

On-device model with LiteRT-LM tool calling later
  -> model may call loadSkill/runJs directly
  -> app still normalizes to the same SkillResult and message types
```

The important rule is that cloud and on-device must receive the same compact skill result for the same user query. Otherwise they will drift and produce different answers.

### Model Input After Skill Execution

When a model pass is needed, the model should receive a constrained payload, not raw unlimited note text:

```json
{
  "userQuestion": "帮我分析所有的记录，都有哪些分类？",
  "skillId": "history-query",
  "intent": "categories",
  "coverage": {
    "timeScope": "all_time",
    "totalCount": 135,
    "processedCount": 135,
    "complete": true,
    "dateRange": {"start": "2026-03-28", "end": "2026-04-25"}
  },
  "facts": {
    "categories": [
      {"name": "应用开发", "recordCount": 42, "summary": "..."}
    ]
  },
  "responseRules": {
    "noAutoSourceLinks": true,
    "noDefaultEvidenceSection": true,
    "noDefaultNextSteps": true,
    "preserveMarkdownParagraphs": true
  }
}
```

This is the main fix for the repeated chat problems:

- exact statistics are not guessed by the model
- source links are opt-in
- "依据" and "下一步" are not default sections
- Markdown structure is explicit rather than post-processed from a single long line

### Stage 1: Skill Selection

Edge Gallery puts selected skill names and descriptions into the system prompt through `SkillManagerViewModel.getSystemPrompt(...)`.

MindFlow should do the same:

```text
Available skills:
- history-query: Query and visualize MindFlow historical notes.
```

In review chat:

- manual skill chip can force a skill
- otherwise the planner/model chooses from the skill summary list

### Stage 2: Skill Instruction Loading

Edge Gallery does not put full skill instructions in the initial prompt. The model calls `loadSkill(skillName)` to load the full `SKILL.md`.

MindFlow should preserve this separation:

- initial context contains skill name and description
- full instruction loads only when the skill is selected

This keeps prompts shorter and makes future skill expansion cleaner.

### Stage 3: Skill Execution

Edge Gallery executes JS through:

```text
runJs(skillName, scriptName, data)
  -> CallJsAgentAction
  -> hidden WebView
  -> ai_edge_gallery_get_result(data, secret)
  -> CallJsSkillResult
```

MindFlow should keep the same outer shape:

```text
runJs(skillId, scriptName, data)
  -> MindFlowJsSkillAction
  -> hidden WebView
  -> ai_edge_gallery_get_result-compatible function
  -> MindFlowSkillResult
```

The first skill can use an Edge Gallery-compatible JS entry:

```javascript
window["ai_edge_gallery_get_result"] = async (data, secret) => {
  try {
    const input = JSON.parse(data || "{}");
    const result = await window.mindflowSkill.run(input, window.mindflowBridge);
    return JSON.stringify(result);
  } catch (error) {
    return JSON.stringify({ error: String(error?.message || error) });
  }
};
```

### Stage 4: Native Data Calls

This is where MindFlow differs from Edge Gallery.

Edge Gallery's `query-wikipedia` skill can fetch directly from Wikipedia because its data source is external and public.

MindFlow skills need private app data, so JS should call a native bridge instead of reading data directly:

```text
window.mindflowBridge.call("history.query", input)
window.mindflowBridge.call("history.categories", input)
window.mindflowBridge.call("note.open", input)
```

This is an adaptation of Edge Gallery's tool boundary, not a replacement for it.

### Stage 5: Result Return And Rendering

Edge Gallery parses `CallJsSkillResult`:

```kotlin
data class CallJsSkillResult(
  val result: String?,
  val error: String?,
  val image: CallJsSkillResultImage?,
  val webview: CallJsSkillResultWebview?,
)
```

MindFlow should begin with the same result categories:

- `result`
- `error`
- `image`
- `webview`

Then review chat appends the result to the message stream:

- text answer as normal assistant message
- webview as a card message
- image as image message
- errors as info/error message

## End-to-End Review Chat Flow

### Example 1: Count Query

User:

```text
我总共有多少条记录？
```

Flow:

```text
ReviewChatHost
  -> Planner selects history-query
  -> JS skill calls bridge.history.count
  -> Native returns exact count and date range
  -> SkillResult.result returns text
  -> UI appends ReviewChatTextMessage
```

This is the MindFlow equivalent of an Edge Gallery JS skill returning only `result`.

### Example 2: Category Summary With Card

User:

```text
帮我看一下所有的记录，帮我总结一下它们都是什么类型的
```

Flow:

```text
ReviewChatHost
  -> Planner selects history-query with intent=categories
  -> JS skill calls bridge.history.categories
  -> Native returns category candidates and counts
  -> JS returns SkillResult.result and SkillResult.webview
  -> optional model pass summarizes compact category result
  -> UI appends ReviewChatTextMessage + ReviewChatWebViewMessage
```

The model does not classify from a small random snippet set. It sees the skill result.

### Example 3: Open Analysis

User:

```text
最近两周我反复纠结的问题是什么？
```

Flow:

```text
ReviewChatHost
  -> Planner selects history-query
  -> JS skill calls bridge.history.query with recent range
  -> Native returns relevant records
  -> Native returns compact batches and coverage metadata
  -> Model composes final analysis
  -> UI appends ReviewChatTextMessage and optional ReviewChatWebViewMessage
```

This uses skill execution for retrieval, but still lets the model handle analysis.

### Example 4: Random Or Low-Intent Input

User:

```text
test
```

Flow:

```text
ReviewChatHost
  -> Planner confidence is low
  -> returns clarification
  -> UI appends ReviewChatTextMessage
```

No skill should run.

## JS And Model Ownership Boundary

The JS skill can:

- call native bridge APIs
- arrange data for visual presentation
- build HTML cards through local skill assets
- request model summarization through approved bridge APIs
- return typed skill results

The JS skill should not:

- directly access Room
- decide unrestricted native actions
- silently open records
- bypass permission checks
- return unbounded raw note contents into model prompts

The model can:

- choose a skill
- generate skill arguments
- summarize compact skill results
- explain visual card output

The model should not:

- be responsible for exact counts
- infer date ranges from partial samples
- decide source links unless explicitly requested
- produce HTML directly for core cards in the first version

## Answer Composition Policy

The default assistant answer should be simple:

```text
direct answer

optional short bullets or table

optional card below the text
```

Do not add these sections by default:

- `依据`
- `下一步`
- `相关链接`
- `命中的记录`

Only add them when the user asks for them:

- user asks "依据是什么" -> add evidence
- user asks "下一步怎么办" -> add next steps
- user asks "给我原始链接/记录" -> add record links
- user asks "列出来" -> add matching records

This policy is enforced in two places:

1. `SKILL.md` tells the model not to add default evidence, next steps, or links.
2. `responseRules` in the model input repeats the same constraints for every model pass.

The app should not rely on post-processing to delete unwanted sections. The model should be given a smaller, clearer job.

### Markdown And Layout Policy

The model response must preserve Markdown structure:

- one blank line between paragraphs
- one bullet per line
- table rows must each be on a separate line
- no mixed `-` and `*` list markers in the same list
- no source links unless requested

The UI renderer should preserve line breaks from the model response. It should not collapse all whitespace into one line before markdown rendering.

## Host Responsibilities

The review chat host owns:

- message lifecycle
- input state
- streaming/status display
- skill result message insertion
- provider fallback
- retry behavior
- persistence of text/webview/image/progress messages

The skill runtime owns:

- loading skill packages
- executing JS
- enforcing timeout
- enforcing bridge permissions
- returning normalized `SkillResult`

The native bridge owns:

- data correctness
- security boundaries
- stable schemas
- conversion from Room/domain models to JSON-safe DTOs

## Native Tool Bridge

JS does not access Room or app internals directly. It calls a restricted native bridge.

Initial API set:

```text
history.count(input)
history.query(input)
history.timeline(input)
history.categories(input)
history.getRecords(input)
note.open(input)
```

## History Data Completeness Contract

The native bridge must make data coverage explicit. This is not optional.

For every history-facing skill result, the bridge returns:

```json
{
  "coverage": {
    "timeScope": {"type": "all_time"},
    "totalCount": 135,
    "matchedCount": 135,
    "processedCount": 135,
    "complete": true,
    "dateRange": {"start": "2026-03-28", "end": "2026-04-25"},
    "nextCursor": null
  }
}
```

Rules:

- exact count, date range, and matched count are always native-computed
- model output must never override native counts
- if `complete` is false, the answer must say it is partial or continue paging before answering
- `limit` means page size, not total search scope
- full-record questions must process all matching records through paging or native aggregation
- category and timeline questions must not use a random top-N sample as the whole truth

### Full-Record Processing Pattern

For all-time summaries and categories:

```text
1. Native computes exact coverage.
2. JS requests records page by page or asks native for aggregation.
3. JS builds compact facts with processedCount.
4. If model summarization is needed, it only sees compact facts and selected summaries.
5. Final answer includes no unsupported global claim unless coverage.complete is true.
```

This is the replacement for the current weak path where a planner retrieves a small set and the model speaks as if it saw everything.

## Review Chat Integration

First target is review chat.

### Current Issue

Review chat currently owns too many responsibilities:

- query planning
- retrieval
- summary construction
- answer structure
- UI fallback behavior

Skill integration should move query execution and result visualization out of the planner.

### First Review Chat Skill

Skill id:

```text
history-query
```

Responsibilities:

- answer history count/list/category/timeline queries
- call native bridge for exact data
- render rich HTML result when useful
- return a short text summary for the chat message

Review chat should still own:

- user message state
- provider selection
- streaming status
- final assistant response composition under the Answer Composition Policy

### Invocation Mode

Review chat uses mixed invocation:

- automatic skill planning by default
- manual skill chips for correction and discovery

Initial manual chips:

- `查历史`
- `分类总结`
- `时间线`
- `生成卡片`

## Execution Status UI

Normal mode:

- `正在查询历史...`
- `正在生成卡片...`
- `正在整理回答...`

Debug mode:

- skill id
- native calls
- query scope
- matched records
- result type
- duration
- error details

This mirrors Edge Gallery's progress panel idea, but keeps ordinary product UI less technical.

## Security And Permissions

### Local Built-In Skills

First version only ships built-in MindFlow skills.

### Remote Official Skills

Future remote official skills require:

- HTTPS transport
- manifest hash
- signature verification
- version pinning
- rollback support

### Network

JS skills are network-capable by default, but manifest must declare network usage.

The runtime should expose network usage in debug/details UI.

### Native Bridge

Native APIs are allowlisted per skill. A skill declaring `history` permission cannot automatically call every native API. It can only call APIs explicitly listed in `nativeApis`.

### WebView Boundary

MindFlow should copy Edge Gallery's local asset loading pattern:

```text
https://appassets.androidplatform.net/skills/{skillId}/...
```

Runtime rules:

- execute JS skills in a hidden execution WebView
- render returned cards in a separate display WebView message
- expose `mindflowBridge` only to the execution WebView
- do not expose private bridge APIs to display-only WebView cards
- resolve relative `webview.url` against the skill asset root
- block navigation outside the allowed skill URL root unless explicitly opened by the app
- block `file://` and arbitrary local paths
- enforce timeout per JS run
- attach a per-invocation token so stale JS cannot call bridge APIs after completion

This keeps the Edge Gallery flexibility while reducing the risk of a visual card reading private notes or triggering native actions.

### Bridge Authorization

Every bridge call must check:

```text
skillId
invocationId
apiName
manifest.nativeApis
time budget
payload size
```

Bridge calls should return structured errors:

```json
{
  "error": {
    "code": "permission_denied",
    "message": "history.getRecords is not allowed for this skill"
  }
}
```

The JS skill can show a friendly failure, but the app logs the structured error in debug mode.

## UI Direction

MindFlow should borrow these Edge Gallery interaction ideas:

- empty state as a scenario launcher
- skill chips as secondary controls
- execution status as visible but compact feedback
- WebView cards embedded in the flow
- skill manager hidden behind a secondary surface

Do not copy:

- developer-heavy wording
- full skill marketplace UI in the first version
- forced session reset whenever skills change

## Engineering Review Lock-In

The first implementation is deliberately smaller than the full platform vision.

Build only one closed loop first:

```text
ReviewChat question
  -> built-in history-query SkillInvocation
  -> native history bridge computes exact coverage and compact facts
  -> existing ReviewChatPlanner sends the compact result to cloud or on-device model
  -> ReviewChat UI renders text, optional record references, and later WebView cards
```

First version scope:

- built-in `history-query` only
- no remote skills
- no user-imported skills
- no graph migration
- no today-card migration
- no JS-exposed `model.summarize`
- no third-party network skill

Implementation principle:

- reuse `ReviewChatPlanner`, `ReviewChatResponsePolicy`, `ReviewChatCorpusQueryEngine`, and current review chat tests
- add a skill-shaped adapter around history querying
- avoid building a parallel chat stack

The broader skill runtime stays in the design, but the first milestone must prove that history answers become more correct before expanding the platform.

## First Implementation Plan

### Step 0: Contract Tests First

Before wiring the UI, add contract tests or local debug fixtures for:

- JS skill returns valid `result/error/image/webview`
- relative `webview.url` resolves to an appassets URL
- unauthorized bridge API is rejected
- timeout returns a structured error
- `history.query` paging reports complete and partial coverage correctly
- Markdown output preserves paragraphs, list lines, and table rows

### Step 1: Platform Skeleton

Add:

- `SkillManifest`
- `SkillRegistry`
- `SkillResult`
- `NativeToolBridge`
- `JsSkillExecutor`

Use bundled internal skills only.

### Step 2: Review Chat Skill

Add built-in `history-query` skill.

The first implementation should support:

- count
- list
- category summary
- timeline anchor
- simple HTML card result

The first skill must include:

- `SKILL.md`
- `scripts/index.html`
- at least one local card file under `assets/`
- `mindflow.skill.json`

### Step 3: Review Chat Host

Update review chat to:

- route eligible requests through `history-query`
- show short execution status
- render skill WebView result in the message stream
- keep fallback to current model-only answer path
- pass compact skill results to both cloud and on-device models through the same payload shape
- persist typed messages, including WebView result metadata

### Step 4: Graph Skill

After chat validates the platform:

- wrap the existing graph HTML/WebView as `concept-graph-view`
- move graph rendering toward the same skill result container

### Step 5: Future Cards

When daily/home cards are redesigned:

- implement them as skill-generated WebView cards
- reuse the same native bridge and skill result protocol

## Acceptance Cases

The first implementation is not complete until these user-facing cases pass.

### A. All-Time Count

User:

```text
我一共有多少条记录？
```

Expected:

- answer uses exact native count
- answer includes correct date range if available
- no source links unless requested
- no `依据` or `下一步` sections

### B. All-Time Categories

User:

```text
帮我分析所有的记录，都有哪些分类？
```

Expected:

- processes all matching records or clearly says the result is partial
- categories are complete enough to cover the dataset, not just one or two records
- each category appears as one bullet or one table row
- no duplicated prefix like `类别：类别：应用开发`
- no automatic source links

### C. Specific Date Or Weekend Query

User:

```text
看一下本周末记录了哪些信息，都有哪些类别
```

Expected:

- date scope is resolved before querying
- all matching records in that scope are counted
- category summary only refers to records in that scope
- no unrelated global knowledge base text is injected

### D. Open Analysis

User:

```text
我记了哪些人生建议？帮我总结成几句话。
```

Expected:

- answer is a concise summary
- no default `依据`
- no default `下一步`
- if examples are included, they are short and directly relevant

### E. Markdown Formatting

Use a response containing:

```markdown
类别：
- 应用开发：...
- 技术优化：...
- 个人成长：...

| 日期 | 记录 |
| --- | --- |
| 2026-04-13 | MindFlow开发问题复盘 |
```

Expected:

- bullets remain separate lines
- table rows remain separate lines
- UI does not collapse the whole answer into one line
- `-` and `*` are not mixed as the same nested list unless intentionally nested

### F. Source Links Are Opt-In

User:

```text
帮我总结一下最近的技术问题
```

Expected:

- no record links

User:

```text
把相关原始记录链接给我
```

Expected:

- links are shown as `ReviewChatRecordRefMessage`
- links point only to records used in the answer

### G. Provider Consistency

Run the same query on cloud and on-device:

```text
帮我看一下所有的记录，帮我总结一下他们都是什么类型的
```

Expected:

- both providers receive the same compact `SkillResult`
- exact counts and date range match
- wording may differ, but facts must not differ

## Open Questions

1. Where should remote official skill metadata be hosted?
2. Which signing format should be used for remote skill bundles?
3. How much bridge data should be cached for offline use?
4. Should generated skill cards be persisted as snapshots or regenerated on every open?

Resolved for first version:

- Execution WebViews are isolated per invocation.
- Display WebViews do not receive private bridge APIs.
- Remote official skills are out of scope for the first implementation.

## Recommended Next Move

Build the platform skeleton and the `history-query` internal skill first, but start with the contract tests above.

This validates:

- JS/WebView skill execution
- native bridge calls
- skill result rendering in chat
- debug status visibility

It also keeps the first milestone small enough to test without waiting for the graph and daily cards redesign.
