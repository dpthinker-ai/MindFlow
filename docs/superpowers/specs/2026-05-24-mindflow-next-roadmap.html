MindFlow 下一步优化 RoadMap

创建时间：2026-05-24 21:15 +08:00。用途：作为下一阶段开发 Spec，承接云侧 AI 接入、回看聊天修复和产品方向复盘后的优化事项。

总体判断

下一阶段不优先增加新的 AI 花样，而是先把“用户提问 → 系统理解 → 取材 → 回答 → 沉淀”的链路做可信、可解释、可继续推进。近期暴露的问题都指向同一个核心：用户问的是自然语言，系统内部却经过时间范围、状态过滤、端/云路由、结构化输出和历史摘要等多层转换；任何一层不透明，用户都会感觉回答不可靠。

因此 RoadMap 的第一优先级是回看/查询可信度，其次是 Flow 的孵化动作化，再往后才是云侧诊断和 AI 产物沉淀的系统化。

P0 回看/查询可信度专项：让每个回答都能解释“我查了什么、用了什么范围、为什么这么答”。
P1 Flow 孵化动作化：从信息展示变成“值得继续养”的行动流。
P2 云侧可诊断与产物沉淀：让多模型、云侧调用和 AI 输出成为可追溯资产。

设计原则

产品原则

  • MindFlow 是 idea incubator，不是普通 AI 笔记，也不是系统状态 dashboard。
  • 捕捉优先：记录入口必须始终短、快、明显，不能为了后续整理要求用户先结构化。
  • 孵化优先于检查：产品应该推动 rescue、expand、collide、verify、reuse，而不是只展示库存。
  • 本地优先：本地知识层是长期记忆源；云侧是增强路径，必须可见、可解释、可追踪。
  • 一个 AI 产品角色：捕捉辅助、记忆维护、线索发现、碰撞生成、资产成形应该像同一个系统在工作。

移动端体验原则

  • 首屏要直接告诉用户“现在有什么值得继续推进”，避免用户在多个页面间记忆上下文。
  • 高频动作放在可触达区域，并保持可见，不把核心操作藏在长按、二级菜单或顶部角落。
  • 每次 AI 等待都要有明确进度/取材状态；空白等待会被用户理解为卡住。
  • 所有状态不能只靠颜色区分；触摸目标按 Android 48dp 以上设计。

阶段 1：回看/查询可信度专项

优先级 P0 建议先做 影响面：回看聊天、查询、历史摘要

目标

让用户每次得到回答时，都能知道系统理解了什么、检索了什么、过滤了什么，以及结果为什么可信。重点不是暴露技术细节，而是用轻量、人话的方式把“理解和取材范围”说清楚。

功能范围

  • 在回看聊天回答顶部增加轻量范围说明，例如：本周 · 18 条记录 · 包含更新记录 · 本地检索。
  • 范围说明至少包含：时间范围、命中记录数、是否包含更新记录、状态过滤、端侧/云侧来源。
  • 当命中为 0 时，必须说明是“没有符合过滤条件”还是“时间范围内没有记录”。
  • 当云侧参与时，沿用低频通知策略，但当前回答内也要有轻量可见标识。
  • 历史摘要保存时继续使用人类可读 preview，禁止再次把结构化 JSON 直接暴露给用户。

验收标准

  • 提问 本周我记录了哪些内容？ 时，回答能显示本周范围、命中数，并能包含本周更新过的旧记录。
  • 提问 最近一个月有哪些未推进的想法？ 时，回答能显示最近 30 天范围和 IDEA 状态过滤。
  • 当没有结果时，UI 不只说“没有记录”，而是说明使用了哪个范围和哪个过滤条件。
  • 回看历史列表与最新回看卡片不出现 {"summary": ...} 或其他原始 JSON。
  • 新增 parser、context packet、formatter 和 UI 层回归测试。

阶段 2：真实问法回归集

优先级 P0 阶段 1 同步推进 影响面：测试体系

目标

把用户真实使用中暴露过的问题，沉淀成长期回归集。以后任何 parser、检索、云侧路由、格式化改动，都必须先守住这些真实问法。

首批问法

  • 本周我记录了哪些内容？
  • 最近一个月有哪些未推进的想法？
  • 我最近在关注什么？
  • 哪些想法一直没动？
  • 最近两周的矛盾帮我串一下

验收标准

  • 每个真实问法都有 parser 级测试，验证时间范围、状态过滤、实体词和意图。
  • 每个真实问法至少有一个 context packet 测试，验证实际取材集合。
  • 关键问法覆盖最终展示摘要，避免再次出现 JSON、空结果误报或范围误判。
  • 新增测试命令记录在开发文档中，作为后续回归固定入口。

阶段 3：Flow 孵化动作化

优先级 P1 阶段 1 稳定后启动 影响面：Flow 首页、记录详情、回看入口

目标

把 Flow 从“信息展示面”变成“今天有哪些东西值得继续养”的行动流。用户打开 Flow 后，不应该像看报表，而应该看到几个可继续推进的思想对象。

建议信息架构

  • Fragile Spark：脆弱但值得救的想法，动作是“继续养”。
  • Recurring Thread：反复出现但尚未命名清楚的线索，动作是“命名它”。
  • Collision：两个旧想法之间值得碰撞的地方，动作是“撞一下”。
  • Asset：已经成熟、可复用的内容，动作是“整理成方案”。
  • Gap：缺证据、缺案例、缺下一步的洞，动作是“找证据”。

验收标准

  • Flow 首屏不使用 库存、当前知识、维护状态 等后台化语言。
  • 每张 Flow 卡片至少有一个明确动作，且动作会把用户带到可继续推进的上下文。
  • 卡片解释为什么值得继续推进，而不只是展示一段摘要。
  • 空状态能引导用户记录一条 Spark，而不是仅显示“暂无数据”。
  • 新增 Flow 卡片生成/排序的单元测试，避免低价值内容占据首屏。

阶段 4：云侧 AI 可诊断

优先级 P1 影响面：设置、云侧调用、用量记录

目标

云侧 AI 设置不只是“能配置”，还要“能解释”。用户应该知道当前 provider 是否可用、最近一次失败原因是什么、今日用量来自哪里、自动模式为什么选择端侧或云侧。

功能范围

  • 每个供应商页面增加“测试连接”动作，展示成功、鉴权失败、网络失败、模型不可用等人话原因。
  • 设置页展示最近一次调用：provider、model、任务、时间、成功/失败、Token 数。
  • 自动模式增加“为什么这样选择”的解释，例如“该任务包含敏感内容，已保持端侧”。
  • 用量统计按 provider、model、任务维度可展开，但默认保持低噪音。

验收标准

  • DeepSeek、智谱、OpenAI、Custom 均可独立测试连接，不互相污染配置。
  • 错误提示可行动，例如提示检查 API Key、Base URL、模型名或网络。
  • 后台云侧使用仍遵守低频通知，不因诊断信息变成通知轰炸。
  • 用量记录不包含 prompt、笔记正文、API Key 或本地文件路径。

阶段 5：AI 产物沉淀回本地记忆

优先级 P2 影响面：回看聊天、Flow、知识层、记录详情

目标

AI 回答不应该只停留在聊天气泡里。好的回答要能被一键沉淀为 Thread、Collision、Asset 或 Gap，并带着来源和生成路径回到本地知识层。

功能范围

  • 回看回答提供“沉淀为 Thread / Asset / Gap”的轻量动作。
  • 沉淀对象记录 provenance：来源记录、生成时间、端侧/云侧、provider/model。
  • 沉淀内容可编辑，默认不自动写入不可逆长期记忆。
  • Flow 可以读取这些沉淀对象，并在后续继续推进时复用。

验收标准

  • 用户可以从一条回看回答创建一个可编辑 Asset 草稿。
  • Asset 草稿能看到来源记录列表，不丢 provenance。
  • 沉淀后的对象能在 Flow 或相关详情中再次出现。
  • 云侧生成的沉淀对象明确标记云侧来源，但不泄露敏感 payload。

建议实施顺序

顺序 主题 理由 完成信号
1 回看/查询可信度专项 直接解决近期用户不信任问题，是后续所有 AI 能力的地基。 真实问法不再误报空结果，回答能显示范围和取材说明。
2 真实问法回归集 把最近踩过的坑固定下来，防止后续功能开发反复回退。 parser、context packet、展示层都有真实问法测试。
3 Flow 孵化动作化 把产品体验从“回看旧内容”推向“继续推进想法”。 Flow 首屏出现 Spark、Thread、Collision、Asset、Gap 类型行动卡。
4 云侧 AI 可诊断 多供应商接入后，配置正确性和路由解释会变成长期维护成本。 每个 provider 可独立测试连接，并能解释最近一次调用状态。
5 AI 产物沉淀回本地记忆 让 AI 回答成为可复用资产，而不是一次性聊天结果。 回看回答可沉淀为带 provenance 的 Thread / Asset / Gap 草稿。

非目标

  • 本 RoadMap 不要求立即增加新的云侧模型协议，例如 Anthropic/Gemini 原生协议。
  • 不把 Flow 改成系统维护 dashboard，不展示后台维护术语作为主体验。
  • 不让后台云侧任务静默升级；云侧使用仍必须可见、低频、可回看。
  • 不在阶段 1 中重写整个知识层，只围绕回看/查询可信度做收敛改进。

验证要求

  • 代码级验证：每阶段至少包含聚焦单测；影响回看/设置/Flow 时补充对应 UI state 或 ViewModel 测试。
  • 发布验证：涉及用户可见体验时，必须构建 release APK：./gradlew --no-daemon :app:assembleRelease。
  • 真机/模拟器安装必须使用 app/build/outputs/apk/release/app-release.apk，不得安装 debug APK 覆盖 release 数据。
  • 禁止执行 adb uninstall、pm clear、AVD wipe 或删除本地模型/数据，除非用户明确授权。
  • 每次实现都要同步更新 implementation-notes.html 或后续对应实现记录，写清楚决策、取舍、验证和安装结果。
