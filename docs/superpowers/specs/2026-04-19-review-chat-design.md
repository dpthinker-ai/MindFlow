# 回看聊天功能设计

## 背景

当前 `回看` 页已经承担了“重新理解旧积累”的职责，但交互仍然是卡片式浏览：

- `沉淀结果` 帮用户看到最近真正成立的结论
- `断层补洞` 帮用户看到值得重新接上的方向
- 底部 `搜索记录` 仍然是一个独立入口，语义上更偏手动搜索，而不是基于历史信息继续讨论

同时，项目已经具备两类可复用的 AI 能力：

1. `云侧 AI`，适合长上下文和综合归纳。
2. `Gemma 4 端侧 AI`，适合离线兜底和受限场景。

这次要补的不是一个泛聊天页，而是一个 `基于历史信息的回看聊天能力`：

- 用户可以基于自己的历史记录提问
- 用户可以在回答基础上继续追问
- 能力重点是 `归纳串联 -> 讨论推进 -> 回忆查找`
- 这条链要和现有回看页语义一致，而不是把一个通用聊天机器人硬塞进来

## 目标

1. 在 `回看` 页增加一个自然的聊天入口，用来基于历史信息提问和讨论。
2. 支持多轮聊天，而不是一次性问答。
3. 检索范围同时覆盖：
   - 原始记录
   - 已沉淀结构
4. 在 `自动` 模式下优先保证回答质量和交互流畅度。
5. 会话默认不长期保存，只有用户显式点击保存时才落库。
6. 让用户能看清本次回答实际由 `云侧` 还是 `端侧` 完成。

## 非目标

1. 不把聊天和现有搜索页合并成统一页面。
2. 不把聊天做成通用开放世界问答。
3. 首版不提供来源卡、证据列表或引用展开。
4. 首版不提供回答后的快捷动作：
   - 打开相关记录
   - 跳线程
   - 新建跟进记录
5. 首版不做多会话管理页。
6. 首版不做自动保存、退出前挽留保存、草稿恢复。
7. 首版不引入向量库、embedding 检索或额外在线检索服务。

## 产品决策汇总

这是本次设计中已经明确的产品决策，后续实现必须遵守：

1. 聊天同时支持：
   - 基于历史信息的查询
   - 基于历史信息的讨论
2. 检索范围选择 `原始记录 + 已沉淀结构`。
3. 聊天入口采用 `回看页内嵌入口卡`，不是独立入口页。
4. 用户一旦发送问题，立即进入 `全屏聊天页`。
5. 入口卡不展示建议问题，只保留：
   - 标题
   - 一句短说明
   - 输入框
6. 保存策略是 `按需保存`，由用户显式触发。
7. 首版不提供聊天结果后的快捷动作。
8. 能力优先级固定为：
   - `归纳串联`
   - `讨论推进`
   - `回忆查找`

## 信息架构

### 1. 回看页

`回看` 页仍然保留现有两张主要卡片：

- `沉淀结果`
- `断层补洞`

底部现有的 `搜索记录` 卡被替换为新的聊天入口卡：

- 标题：`和历史聊聊`
- 说明：`基于你的历史记录和沉淀内容继续聊`
- 一个输入框

这张卡不承载消息流，也不展示预设问题。它的唯一职责是：

- 让用户明确知道这里可以问历史相关的问题
- 接住第一句输入
- 把用户带入专用的回看聊天页

### 2. 回看聊天页 `ReviewChatScreen`

用户一旦提交问题，就从回看页进入全屏 `ReviewChatScreen`。

这个页面负责：

- 展示当前会话的消息流
- 接住后续追问
- 展示当前回答的 provider 信息
- 提供显式保存入口

它不负责：

- 展示来源卡
- 承担复杂搜索筛选
- 直接发起其他业务动作

### 3. 搜索页保持独立

现有搜索页继续负责：

- 条件筛选
- 关键词检索
- 列表浏览

聊天功能不并入搜索页。原因很直接：

- 搜索是在“找一条记录”
- 回看聊天是在“基于历史记录继续讨论”

两者底层都可以复用本地数据，但状态机和用户心智不同，不能混成一个入口。

## 页面流转

### 主路径

1. 用户进入 `回看` 页。
2. 用户在底部聊天入口卡输入问题。
3. 用户点击发送。
4. 页面进入 `ReviewChatScreen`。
5. 页面先展示用户问题，再展示 AI 回答。
6. 用户继续追问，形成多轮会话。
7. 用户如果觉得这段对话有价值，再点击 `保存`。

### 返回路径

- 用户从聊天页返回回看页时：
  - 如果本次会话未保存，默认直接丢弃
  - 不弹“是否保存”挽留框

### 保存后的最小访问路径

首版不做完整的会话管理页，但保存不能是黑洞。

因此，首版提供一个最小可达路径：

- 如果存在已保存对话，回看页入口卡下方显示一条次级文本入口：
  - `查看最近保存的对话`
- 点击后打开最近一次保存的会话
- 首版只保证“最近一次保存的会话可再次打开”
- 首版不提供完整历史列表、搜索或批量管理

这个约束保证：

- 保存是有意义的
- 范围仍然控制在首版可落地的大小

## 能力边界

### 优先能力

#### 1. 归纳串联

系统优先回答：

- 最近两周反复出现了哪些矛盾
- 哪些记录其实指向同一条主线
- 工作和副项目里哪些主题在互相呼应

#### 2. 讨论推进

系统其次回答：

- 为什么当前更值得推进某个方向
- 一条判断背后真正的分歧是什么
- 基于过去记录，接下来更值得怎么想

#### 3. 回忆查找

系统最后回答：

- 之前什么时候提过某个问题
- 哪些历史记录和当前话题最接近

### 不支持的问法

以下类型不作为首版目标：

- 开放世界知识问答
- 实时联网查询
- 与历史信息无关的闲聊
- 需要多工具联动执行的任务型 agent 行为

## 检索范围

### 1. 原始记录

原始记录包括：

- 标题
- 正文
- 标签
- 状态
- 文件夹
- 创建/更新时间
- 归档记录

首版默认允许检索归档记录，因为这是“基于历史积累”的能力，而不是只看活跃记录。

### 2. 已沉淀结构

首版纳入的结构化来源包括：

- `WeeklyReviewState` 的周回看结果
- `FollowedDirectionSummary` 的方向/线程摘要
- `LocalKnowledgeMaintenanceSnapshot` 的当前判断、最近吸收、开放问题等压缩结果
- `FlowKnowledgeCompressionState` 的当前主线、沉淀和缺口压缩
- `DirectionWikiCoordinator` / 知识层里可检索的知识对象摘要
- 概念图谱或知识层中已经生成的概念/关系摘要

### 3. v1 检索策略

首版不引入向量库。检索采用本地规则组合：

- 关键词命中
- 标签命中
- 主题/线程 key 命中
- 时间范围线索
- 最近对话上下文中的延续词

然后将结果裁成一个有上限的上下文包，再交给模型生成回答。

这意味着首版是：

- `规则检索 + AI 生成`

而不是：

- `全量历史直喂模型`
- `embedding / vector search`

## 回答生成链路

### 步骤 1：问题理解

系统先解析用户问题，识别：

- 它更偏 `归纳串联 / 讨论推进 / 回忆查找` 哪一类
- 是否带时间范围
- 是否带主题词、项目词、状态词
- 是否是对上轮回答的追问

### 步骤 2：候选材料检索

系统从两类来源中取候选材料：

- 原始记录候选
- 结构化摘要候选

### 步骤 3：上下文组装

系统把候选材料压成有限上下文包，包含：

- 问题类型
- 当前问题
- 近期会话摘要
- 原始记录摘录
- 结构化摘要摘录

上下文包必须有限，不允许把全量原始记录直接交给模型。

### 步骤 4：回答生成

模型根据上下文包生成自然语言回答。

首版回答风格要求：

- 优先清楚、具体
- 不要像泛聊天鼓励
- 不要写成工作汇报
- 回答要像“真的读过你的历史信息后给出的判断”

### 步骤 5：会话摘要更新

每轮回答后，系统生成一个轻量会话摘要，用于后续追问续接，而不是每次把整段完整历史重新拼回 prompt。

## Provider 路由策略

### 全局模式仍然存在

功能仍然遵守全局 AI 执行模式：

- `自动`
- `仅端侧`
- `仅云侧`

### 回看聊天的自动模式例外

对于回看聊天，这次明确规定：

- `自动` 模式下，所有回答回合都走 `云侧优先 -> 端侧回退`

原因：

1. 这类任务更依赖长上下文理解和综合归纳能力。
2. 回看聊天是强交互场景，首要目标是回答质量和顺畅度。
3. 端侧更适合作为：
   - 用户主动选择 `仅端侧`
   - 云侧不可用时的兜底

这和编辑页 AI 的交互型任务策略保持一致，避免再次出现“自动模式其实总是先跑端侧，整机卡顿”的问题。

### UI 必须显示实际 provider

每条 AI 回答都必须显示本次实际 provider，例如：

- `本次由云侧完成`
- `云侧不可用，已回退端侧`
- `本次由端侧完成`

这个信息不需要很重，但必须可见。

## UI 结构和状态

### 回看页入口卡

入口卡仅保留：

- 标题
- 一句说明
- 输入框

不保留：

- 预设问题 chip
- 消息流
- 复杂筛选

### 聊天页

聊天页结构：

- 顶部栏
  - 返回
  - 当前会话标题
  - `保存`
- 中部消息流
  - 用户消息
  - AI 回答
  - provider 标识
- 底部输入区
  - 输入框
  - 发送按钮

AI 回答视觉上更接近安静的阅读块，不做强 IM 气泡感。

### 状态

#### 初始态

- 回看页只有入口卡

#### 发送中

- 进入聊天页
- 用户问题立即出现
- AI 回答区显示明确生成中状态

#### 成功态

- 展示完整回答
- 允许继续追问

#### 失败态

- 展示简洁失败说明
- 提供 `重试`
- 自动模式下如果发生 provider 回退，要把回退信息讲清楚

#### 未保存态

- 默认就是未保存态
- 返回即丢弃

#### 已保存态

- 顶部 `保存` 变为已保存状态
- 再次打开时显示为只读历史会话

## 保存策略

### 保存原则

- 只有用户显式点击 `保存` 才落库
- 不做自动保存
- 不做退出挽留

### 保存内容

每段保存的对话至少包括：

- 会话标题
- 创建时间 / 更新时间
- 完整消息流
- 每条 AI 回答的 provider 信息

首版不保存：

- 检索中间结果
- prompt 原文
- 完整上下文包

### 标题生成

首版会话标题直接基于第一句用户问题生成短标题；如果生成失败，就退回第一句问题的截断文本。

## 持久化模型

首版新增一套独立的本地持久化，不复用普通 note。

建议的数据结构：

- `ReviewChatSession`
  - `id`
  - `title`
  - `createdAt`
  - `updatedAt`
  - `saved`
- `ReviewChatMessage`
  - `id`
  - `sessionId`
  - `role`
  - `content`
  - `provider`
  - `createdAt`

关键约束：

1. 未保存会话只存在于内存态/ViewModel。
2. 保存时才写入本地存储。
3. 已保存会话与普通记录系统隔离，不进入普通笔记流。

首版建议使用独立本地仓储（例如 Room-backed repository）来管理已保存对话，因为消息结构天然是会话-消息关系；不应把它挤进现有 Markdown note 持久化链。

## 架构拆分

首版建议新增三块核心单元：

### 1. `ReviewChatEntryCard`

职责：

- 回看页上的聊天入口 UI
- 接住第一句输入
- 触发进入聊天页

### 2. `ReviewChatViewModel`

职责：

- 持有当前会话状态
- 维护消息流
- 管理发送中/失败/保存状态
- 处理显式保存

### 3. `ReviewChatPlanner` / `ReviewChatCoordinator`

职责：

- 问题分类
- 检索候选材料
- 组装上下文包
- 调用 AI
- 管理 `自动` 模式下的 provider 策略

这条链不能直接复用 `SearchViewModel`，因为：

- 搜索是筛选 + 结果列表
- 回看聊天是检索增强 + 多轮对话 + 显式保存

表面都在“找历史内容”，但状态机完全不同。

## 测试策略

### 单测

覆盖以下逻辑：

1. 问题分类：
   - 归纳串联
   - 讨论推进
   - 回忆查找
2. 检索上下文组装：
   - 原始记录和结构化来源都能进上下文
   - 上下文长度被裁剪
3. provider 路由：
   - 自动模式下确实是 `云侧优先 -> 端侧回退`
4. 保存逻辑：
   - 只有显式保存才落库
   - 未保存返回不落库

### ViewModel 测试

覆盖以下状态流：

1. 发送问题后进入生成中
2. 成功后出现回答
3. 失败后出现错误和重试
4. 多轮追问能带上当前会话摘要
5. 保存后状态切换正确

### UI / 集成测试

至少覆盖：

1. 回看页可以进入聊天页
2. 发送问题后聊天页出现消息流
3. 顶部 `保存` 可点击
4. 保存后返回回看页，入口卡能打开最近保存的对话

## 首版验收标准

如果首版正确落地，应该满足以下标准：

1. 用户在回看页看到的是一个安静明确的聊天入口，而不是另一个搜索框。
2. 首次发问后立即进入全屏聊天页，体验清晰。
3. 回答明显是基于历史信息，不像泛聊天。
4. 用户能继续追问，形成连续会话。
5. 用户能看出这次实际由云侧还是端侧完成。
6. 默认不会产生一堆未保存垃圾会话。
7. 保存后的对话至少能从回看页再次打开最近一次。

## 未来扩展点

这些不是首版目标，但架构要留出空间：

1. 来源卡 / 依据展开
2. 回答后的快捷动作
3. 已保存对话列表
4. 保存后继续续聊
5. 更强的语义检索（例如 embedding / vector search）
6. 更细粒度的端侧/云侧分工
