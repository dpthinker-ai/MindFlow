Cloud AI Runtime Policy Implementation Notes

Created on 2026-05-24 for the DeepSeek/general cloud AI provider implementation.

Initial Decisions

Snapshot before implementation. The original workspace had existing app and test edits plus large untracked local data directories. I committed the app/test code snapshot as e5c4751 chore: snapshot current app changes and excluded data/, recovery/, tmp/, and product/local documents from the snapshot.
Isolated worktree. The implementation is being done in /Users/dpthinker/.config/superpowers/worktrees/MindFlow/cloud-ai-runtime-policy on branch codex/cloud-ai-runtime-policy. This keeps the main workspace's local artifacts and future user edits separate from the Spec implementation.
Implementation boundary. The first production version stays on OpenAI Chat Completions-compatible providers. DeepSeek, OpenAI, Zhipu, and custom providers share one registry and request adapter. Native Anthropic/Gemini protocols and cloud audio/image upload remain out of this change because they need separate explicit consent and payload design.
Audit storage choice. Usage audit will use a local JSONL/file-backed repository instead of a Room migration. This mirrors the existing AiTaskTraceRecorder style, avoids a database migration for metadata-only events, and keeps the release validation lower risk.

Implementation Log

Plan written. The execution plan is stored at docs/superpowers/plans/2026-05-24-cloud-ai-provider-runtime-policy.md. The plan uses TDD checkpoints for provider registry, runtime settings, policy guards, usage audit, settings UI, and release verification.
Provider registry and adapter. Added a generic cloud provider registry with zhipu, openai, deepseek, and custom. DeepSeek defaults to https://api.deepseek.com, /chat/completions, and deepseek-v4-flash, with deepseek-v4-pro available. Deprecated deepseek-chat and deepseek-reasoner are normalized to the current default.
Adapter testing decision. The request adapter returns a JSON string rather than an Android org.json.JSONObject. JVM unit tests use Android's not-mocked org.json stub, so a string renderer keeps request construction testable without adding another JSON dependency or shifting these tests to instrumentation.
Verification. Ran ./gradlew --no-daemon :app:testDebugUnitTest --tests "com.mindflow.app.data.ai.cloud.*"; the provider and adapter tests pass.
Runtime settings split. Added AiRuntimeSettings with explicit execution mode, foreground/background cloud permission, low-frequency notification mode, and background request/token defaults. The first repository is DataStore-backed and can migrate its initial execution mode from the existing on-device settings repository.
Default execution mode correction. Fixed OnDeviceExecutionModeCodec.decode(raw = null, legacyPreferOnDevice = null) to return AUTOMATIC. The older explicit legacy value preferOnDevice = false still returns CLOUD_ONLY, so intentional old cloud-only choices remain intact.
Verification. Ran ./gradlew --no-daemon :app:testDebugUnitTest --tests "com.mindflow.app.data.settings.OnDeviceExecutionModeCodecTest" --tests "com.mindflow.app.data.settings.AiRuntimeSettingsTest"; the runtime/default-mode tests pass.
Task policy registry. Added a first local policy table for router-backed tasks. Foreground polish uses cloud then on-device with Toast notice; background summary/topic/tag/folder/graph tasks use on-device then cloud with aggregated notice; audio, translation, and image understanding remain on-device only.
Sensitivity and budget guards. Added a rule-based sensitivity classifier and a background-only budget guard. High sensitivity background content, such as identity numbers, bank-card-like numbers, credentials, tokens, or similar keywords, blocks cloud fallback. Foreground user-triggered cloud work is not blocked by background request/token budget.
Router compatibility. AiTaskRouter now filters candidate providers through task policy and cloud guards while preserving the existing automaticPreference behavior for current callers. This avoids forcing all planners to be rewritten in the same commit.
Verification. Ran focused policy/router tests including AiTaskPolicyRegistryTest, AiDataSensitivityClassifierTest, CloudUsageBudgetGuardTest, AiTaskRouterPolicyTest, and the existing AiTaskRouterTest; they pass.
Usage audit. Added AiUsageEvent, FileAiUsageEventRepository, and AiCloudUsageReporter. Events include provider, model, task, trigger mode, payload policy, sensitivity, token count, success/failure, and notification batch metadata. Events intentionally do not include prompt text, note body, API keys, attachment paths, or local file paths.
Low-frequency notification aggregation. Added CloudUsageNotificationAggregator with a 5-minute earliest flush, 30-minute max window, and 3-per-day cap. The aggregator returns notification batches and marks events as notified through the repository; Android notification delivery stays outside this pure component.
Visible background notification dispatch. Added CloudUsageNotifier and AndroidCloudUsageNotifier after noticing that aggregation alone would not visibly inform the user. Background batches now flow from AiCloudUsageReporter into an Android notification channel named mindflow_cloud_ai_usage, using the existing notification permission and launcher icon patterns. Foreground cloud calls still use Toast via MindFlowApp.
Direct client audit bridge. AiServiceClient now accepts an optional AiCloudUsageReporter. All cloud HTTP calls made through the client produce usage events, including the planners that still call the client directly. This is an incremental bridge before every planner is moved behind the longer-term runtime orchestrator.
Verification. Ran focused usage and notification tests plus policy/router regression tests; they pass.
Settings and app wiring. Wired AiRuntimeSettingsRepository, FileAiUsageEventRepository, AiCloudUsageReporter, and the DeepSeek-aware AiServiceClient through AppContainer. MindFlowApp now observes foreground cloud notices and shows a Toast.
Settings copy and provider selection. The cloud AI screen now exposes all non-custom built-in providers, including DeepSeek. The description no longer claims background maintenance never uses cloud; it says automatic mode may use cloud by task policy, with low-frequency notification and usage records when content leaves the device.
Execution mode compatibility. Settings changes write the new runtime settings repository and continue writing the old on-device field as a compatibility mirror. This keeps existing code paths and tests stable while routing reads move to runtime settings.
Verification. Ran SettingsCloudAiProviderTest and existing settings execution-mode persistence tests; they pass.
Notification dispatch verification. Added a TDD regression test that first failed because CloudUsageNotifier and backgroundNotifier did not exist, then passed after adding the dispatcher bridge. Ran ./gradlew --no-daemon :app:testDebugUnitTest --tests "com.mindflow.app.data.ai.AiCloudUsageReporterTest"; it passes.
Provider draft overwrite fix. Real-phone validation found that selecting DeepSeek could fall back to the previously saved Zhipu settings while editing. The cause was that SettingsViewModel merged every AiSettingsRepository emission into the whole form, including usage/verification-only refreshes. The fix now preserves the dirty draft's provider, API Key, Base URL, model, and enabled state while still accepting fresh verification and usage metadata.
Independent provider pages. Reworked the cloud AI settings entry into a provider overview plus one page per provider. Opening DeepSeek, Zhipu, OpenAI, or Custom now edits only that provider's form, so the selector is no longer a chip group sharing one live form.
Provider-level persistence. After reviewing the separate-page direction, I was not satisfied with UI-only isolation. PreferencesAiSettingsRepository now stores provider-specific API Key, Base URL, model, enabled flag, verification, and usage counters while preserving the existing active-provider settings for runtime callers. Existing legacy values are used as the initial active provider profile, so current Zhipu settings migrate without a destructive reset.
Provider switching behavior. Switching to a provider with no saved profile starts with an empty API Key and that provider's default Base URL/model. Switching to a provider with a saved profile restores that provider's own values. This avoids carrying a Zhipu key into DeepSeek while still allowing future provider-specific credentials to be remembered.
Provider-scoped usage merge. Phone validation showed that 今日请求, 今日成功, and 今日 Token could still look shared after opening another provider page. The persistence layer already stores provider-level counters, but SettingsViewModel.mergeSavedAiSettings accepted usage and verification refreshes from the active saved provider even while a different provider draft was being edited. The merge now applies usage and verification metadata only when the saved provider matches the visible provider; otherwise the visible provider keeps its own counters until getProviderSettings(providerId) loads that provider's profile.
